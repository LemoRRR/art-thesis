## F1 材料理解 + 学段判断

### 已具备

- Stage1 是统一入口，支持输入题目、想法、大纲、材料，也支持上传 PDF/Word/TXT。
- 已有理解模型字段：`pathType`、`inputType`、`hasDetectedOutline`、`hasDetectedDraft`、`academicLevelSuggestion`、`academicLevelReason`、`coreArguments`、`outlineSummary`、`draftSummary`、`nextStepRecommendation`。
- Prompt 明确要求：已有论文只理解研究对象、核心论点和结构，不学习语言风格。
- 理解完成后会展示结构化卡片，包含路径判断、输入类型、大纲/正文识别、核心论点、AI 学段建议、建议下一步。
- 用户必须选择本科、硕士或期刊后才能进入 Stage2。

### 未完成 / 风险

- 当前会自动推断并写入 `academicLevel`，虽然 UI 允许用户确认，但合同表述是“最终以甲方确认为准”，建议弱化自动写入为“建议值”，由用户点击后才作为确认值。
- 路径识别结果尚未真正驱动后续流程：
  - `confirm_detected_outline` 没有明显进入“提取大纲确认”的专门状态。
  - `revise_existing_draft` 没有把已有论文解析成可编辑 sections 并默认进入修改模式。
- 已有论文路径“不学习语言风格”的边界在 Stage1 prompt 有，但上传资料库后台 `extractDimensions` 仍会提取 `style_extract` 作为资料维度；需要在产品文案和数据使用上避免用户误解为 F3 风格学习。

### 建议补齐

1. 把 `confirmedAcademicLevel` 从建议值中分离出来，只有用户点选后才写入后续生成标准。
2. Stage2 根据 `hasDetectedOutline` / `outlineSummary` 优先展示“确认识别到的大纲”。
3. 已有论文修改路径增加“解析为正文 sections”的流程，Stage3 默认修改，不自动全文重写。

## F2 正文撰写 / 修改

### 已具备

- Stage3 支持一键全文生成，并按大纲逐章生成。
- 已有 `生成当前小节` 按钮；当前小节已有内容时会确认“只会替换当前小节，不影响其他内容”。
- 当前小节生成后会记录版本快照。
- 按意见修改当前激活章节，输出只替换当前 section。
- 收尾生成支持摘要、Abstract、关键词、引言和结语；生成后支持基于原结果继续提交调整。
- Prompt 已接入禁用词数组 `bannedPhrases`，没有配置时使用默认禁用词。
- Stage3 可选择风格档案参与生成和修改。

### 未完成 / 风险

- 合同要求模式 A 输入“二级小标题”，当前实现是“选择当前 section / 章节”，还没有单独的二级小标题输入入口。
- 模式 A / 模式 B 微调不超过 10 次：未见次数统计、超限提示或变更处理逻辑。
- 模式 B 输入修改意见上限 500 字、原文段落上限 5000 字：当前未做硬限制。
- 禁用词规则由甲方提供：当前缺少可编辑配置界面或导入入口。
- “每次只处理当前小节，不自动推进”已经在当前小节生成中成立，但全文生成仍是显著入口；这可以保留为增强能力，但验收时要明确“逐小节能力已存在”。
- 已有论文修改路径尚未默认进入修改模式。

### 建议补齐

1. 增加“按二级小标题生成当前小节”的轻量入口，可以复用现有 `generateActiveSectionOnly`。
2. 给 section/paragraph 维护 `adjustCount` 或本地计数，达到 10 次后提示。
3. 给修改意见和原文段落加前端校验，超限时禁止提交。
4. 增加项目级禁用词配置入口，写入 `project.context.bannedPhrases`。

## F3 语言风格记忆库

### 已具备

- 已新增 `/style-profiles` 页面和侧边导航入口。
- 支持学生风格档案列表、新建、编辑、删除。
- 支持 Word/PDF/TXT 上传；解析后只截取前 10000 字做风格画像。
- 每张风格名片最多 3 个参考文档，档案总数本地截断为 50 个。
- 已有 Supabase `style_profiles` 表、RLS、后端 CRUD API。
- 支持 txt/csv 导出。
- Stage3 顶部和输入框均支持调用风格档案；prompt 只注入画像，不注入原文内容。

### 未完成 / 风险

- “同一学生第二次使用时自动调取上次档案”目前更像列表/选择能力，未见根据学生名自动选中最近档案。
- 档案数上限本地生效，后端 API 没有明确阻止第 51 个写入。
- 风格档案上传会复用资料库上传解析流程，解析产生的 library item 可能留在资料库中；需要确认是否符合“风格库与资料库分开”的产品边界。

### 建议补齐

1. 在创建/选择项目时根据学生名或项目元信息推荐最近风格档案。
2. 后端 `POST /api/style-profiles` 增加 50 个档案上限校验。
3. 风格档案上传解析产生的临时资料，标记为内部用途或不出现在普通资料库列表。

## F4 艺术类量表生成 + 质性研究辅助

### 已具备

- 未发现可直接对应的实现。

### 未完成

- 量表生成：输入研究主题，上限 200 字，输出不超过 20 题。
- 扎根分析：输入文本，上限 5000 字，输出编码树和频次统计。
- 情感编码：输入文本，上限 5000 字，输出编码说明表。

### 建议补齐

2. 三个 Tab：量表生成、扎根分析、情感编码。
3. 新增三组 prompt，先以前端流式调用实现，结果保存在当前会话本地状态。
4. 加输入字数校验、复制按钮、导出 txt/docx 入口。

## F5 案例参考与提取

### 已具备

- 资料库中已有“提取案例”Tab，用户可粘贴参考文献内容并提取案例。
- 保存后会生成 `case` 类型资料。
- 写作上下文会把 `case` 类型资料放入“可用案例参考”，Stage3 生成和修改可调用。
- 上传资料后台会提取 `cases_extract` 维度，作为资料库可复用信息。

### 未完成 / 风险

- 合同要求支持 Word/PDF 单次输入上限 50000 字；当前“提取案例”Tab 主要是粘贴文本，prompt 只截取 6000 字。
- 合同要求每次输出不超过 20 条；当前 prompt 没有硬性 20 条限制。
- 合同要求提取结果当前会话内有效，跨会话不自动保留；当前保存为资料库后会持久化，和合同边界不完全一致。
- 合同写明“调用豆包大模型”；当前案例提取用 `callGPT`。

### 建议补齐

1. 新增“会话案例池”，默认只在当前项目/当前会话内有效。
2. 支持上传 Word/PDF，解析后截取最多 50000 字进入案例提取。
3. prompt 明确“最多 20 条”，结果结构化保存为数组。
4. 如需严格按合同，案例提取改为豆包调用；或在文档中确认 GPT 为兼容实现。

## F6 成果文本导出

### 已具备

- Stage3 支持复制全文。
- Stage3 支持导出 docx。
- docx 会尽量保留标题、正文、脚注、参考文献。
- 风格档案支持 txt/csv 导出。

### 未完成 / 风险

- 合同要求“全部生成内容支持一键复制或导出为 txt/docx”，当前主要覆盖论文正文和风格档案，Stage1 理解模型、Stage2 大纲、F4/F5 工具结果还没有统一导出规范。
- Stage3 未见 txt 导出按钮。
- docx 兼容性免责声明未在导出 UI 中明确呈现。

### 建议补齐

1. 给 Stage1 理解结果、Stage2 大纲、Stage3 全文、F4/F5 结果统一增加“复制 / 导出 txt / 导出 docx”小工具。
2. Stage3 DocumentToolbar 增加 txt 导出。
3. 在导出区域增加兼容性提示文案。

## F7 历史回溯

### 已具备

- AI 生成、当前小节生成、按意见修改、手动编辑、脚注变更等会创建版本快照。
- VersionPanel 支持查看预览和恢复任意历史版本。
- 后端 versions 表支持按项目保存和拉取版本。

### 未完成 / 风险

- 合同要求每篇论文不超过 10 个历史版本；当前前后端均限制为 30 个。
- 合同要求当前版本与任意历史版本简单对比展示；当前只有预览和恢复，未见 diff 对比。
- 合同要求历史记录保存周期 90 天并自动清除；当前未见清理逻辑。
- 合同要求历史记录以会话为单位存储，跨设备同步不在范围内；当前已接入 Supabase 跨设备云端同步，能力更强，但合同边界需要文案说明。
- VersionPanel 底部写的是“最多保留 30 个版本 · 存储在本地浏览器”，与当前后端同步能力及合同 10 个版本不一致。

### 建议补齐

1. 把版本上限改成合同口径 10 个，或保留 30 个但在验收说明中确认这是增强能力。
2. 增加简单 diff：按章节显示“新增/删除/修改”文本差异。
3. 增加 90 天清理逻辑，本地启动时清理，后端查询或定时任务清理。
4. 更新 VersionPanel 说明，避免“本地浏览器”和云端同步冲突。
